背景 · FLINK 集群的两种角色

Job Manager vs Task Manager

Job Manager · "包工头"×1
  • 协调管理，自己不算数据
  • 拆作业为子任务，派发到 TM 的 slot
  • 监控任务状态与 TM 心跳
  • 触发 checkpoint，失败时决定重启策略
  • 日志典型输出：RUNNING → RESTARTING、调度信息
Task Manager · "施工队"×N
  • 实际处理数据的工作进程
  • 跑算子逻辑：map / filter / window / UDF
  • 提供 task slot，并行度越高需要越多
  • 持有算子 state，checkpoint 时上传备份
  • 日志典型输出：UDF 异常栈、TimeoutException / heartbeat

Managed Flink 全托管这两种角色，你只管代码 / 并行度 / KPU · JM 和所有 TM 写同一个 log stream，靠 threadName / logger 分辨谁在说话

固定 JSON SCHEMA

六个关键字段，排查全靠它们

messageType

日志级别，过滤错误：filter messageType="ERROR"

message

正文，异常栈 / 状态变更都在这里，关键词匹配主字段

logger

来源类名，如 PythonDriver、连接器类

locationInformation

代码位置 类.方法(文件:行)，快速定位代码

threadName

线程名，区分 JM / TM、连接器线程

applicationARN

应用 ARN，多应用共用日志时按应用过滤

最容易混淆的两个概念

日志级别 vs 监控级别

日志级别 Logging Level
  • 控制写多少详细度的日志，作用于所有 logger
  • DEBUG / INFO / WARN / ERROR，不支持 per-package
  • 生产建议 INFO/WARN，排查疑难临时开 DEBUG 再调回
监控级别 Metrics Level
  • 控制 CloudWatch 指标粒度，不是日志
  • Application → Task → Operator → Parallelism
  • Parallelism 级别不建议用于并行度 > 64，成本高

第 7 章 · LOGS INSIGHTS QL 语法

管道式流水线：fields | filter | stats

fields

选择要显示的字段：fields @timestamp, message

filter

过滤，支持 = / like / 正则：filter message like /FAILED/

stats + bin

聚合与时间趋势：stats count(*) by bin(5m)

parse

正则抽取临时字段，把非结构化正文变成可聚合的列

sort / limit

排序限行：sort @timestamp desc | limit 20

@ 内置字段

@timestamp / @message / @logStream，JSON 顶层字段自动发现

按扫描量计费 · 先缩小时间窗 · 单次查询最多返回 10000 行

第 7 章 · OPENSEARCH 语法支持

三种查询语言，同一批日志

CloudWatch Logs 原生集成 OpenSearch PPL / SQL —— 日志不动、零迁移，控制台下拉框即可切换

Logs Insights QL
filter messageType="ERROR"| stats count(*) by logger| sort errors desc
OpenSearch PPL
where messageType="ERROR"| stats count() by logger| sort -errors | head 10
OpenSearch SQL
SELECT logger, COUNT(*)FROM kda_demoWHERE messageType='ERROR'GROUP BY logger

PPL 多出 top/rare/dedup/trendline · SQL 支持 JOIN/子查询，且对 AI 生成最友好 · QL 文档示例最多
